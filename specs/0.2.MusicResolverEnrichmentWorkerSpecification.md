# Music Resolver Enrichment Worker Specification

## Context

This specification assumes the following have already been implemented:

1. `Music Resolver API Specification`
2. `Music Resolver Authentication & Abuse Protection Specification`

The API already:

- serves public search requests
- searches only local/cache data in the request path
- records unresolved demand signals
- applies authentication, trust levels, and rate limits
- prevents public traffic from directly calling third-party providers

This document specifies the background enrichment worker.

---

# Goal

Build a C# background enrichment system that turns unresolved demand into enriched music identifier mappings.

The worker must:

- enrich popular songs first
- stage enrichment from cheapest to most expensive
- include Apple/iTunes enrichment
- exclude Spotify initially
- rate-limit third-party music providers
- make provider limits configurable
- avoid public request-path third-party calls
- persist all successful mappings locally
- use Azure-native infrastructure
- follow the same testing style as the API

---

# Non-Goals

Do not implement Spotify enrichment in the initial version.

Do not allow public API requests to directly trigger Apple calls.

Do not implement a full MusicBrainz mirror unless explicitly added later.

Do not use mocking libraries by default.

---

# High-Level Architecture

```text
Public API
  ↓
ResolutionDemand records
  ↓
Enrichment Scheduler
  ↓
Prioritised Azure Service Bus queues
  ↓
Enrichment Workers
  ↓
Mapping Store / Query Cache / Search Index
```

The API records demand. The scheduler decides what work is worth doing. Workers perform enrichment under provider-specific budgets.

# Enrichment Stages

Work must be staged from cheapest to most expensive.

## Stage 0: Existing Local Mapping

Cost: free.

Sources:

- Track mapping table
- Query cache
- local search index

Purpose:

- avoid duplicate work
- resolve demand already enriched by another job

## Stage 1: Local MusicBrainz-Derived Dataset

Cost: very low.

Sources:

- Blob-hosted MusicBrainz dump
- compact derived local index
- local MBID / ISRC / URL mappings

Purpose:

- find ISRC
- find MBID
- find Apple/iTunes URLs already present in open metadata
- extract Apple IDs from known URLs

No live third-party calls.

## Stage 2: MusicBrainz API

Cost: low to moderate.

Purpose:

- resolve artist/title to MBID
- resolve ISRC to recordings
- retrieve external URL relationships
- retrieve missing metadata when local dataset is stale

Must be rate-limited by configurable provider budget.

## Stage 3: Apple/iTunes Enrichment

Cost: expensive / sensitive.

Purpose:

- resolve high-demand, high-confidence candidates to Apple/iTunes IDs
- use only when cheaper stages fail
- run under a strict configurable provider budget

Initial Apple strategies:

```text
ISRC available:
  use Apple Music catalog lookup by ISRC if configured

No ISRC but strong metadata:
  use iTunes Search API fallback with strict budget

Existing Apple/iTunes URL:
  extract ID without external call
```

Apple enrichment must be feature-flagged and budgeted separately from MusicBrainz.

---

# Explicitly Excluded Initial Provider

## Spotify

Do not implement Spotify enrichment in the initial version.

Leave provider interfaces extensible so Spotify can be added later, but no Spotify API calls, credentials, workers, queues, or tests are required initially.

# Priority Model

The scheduler must enrich popular, legitimate, cheap work first.

## Priority Formula

```text
priority =
  demand_count * 5
+ distinct_install_count * 10
+ recent_demand_count * 3
+ known_identifier_bonus
+ high_trust_bonus
- provider_cost_penalty
- abuse_risk_penalty
- previous_failure_penalty
```

## Bonuses

```text
known ISRC:
  +50

known MBID:
  +30

known artist and title:
  +20

attested demand:
  +20
```

## Penalties

```text
MusicBrainz API:
  -10

Apple Music:
  -50

iTunes Search:
  -75

high risk score:
  -100

previous failed attempt:
  -10 per failure
```

The exact values must be configurable.

# Scheduling Behaviour

The scheduler runs periodically, for example every minute.

## Scheduler Flow

```text
Load unresolved/partial demand
  ↓
Filter out blocked/suspicious demand
  ↓
Skip anything before NextEligibleAt
  ↓
Check if already resolved locally
  ↓
Calculate priority
  ↓
Decide next cheapest valid stage
  ↓
Check provider budget
  ↓
Enqueue job
```

## Scheduling Rules

- Never enqueue duplicate active jobs for the same query and stage.
- Prefer cheaper stages before Apple.
- Do not enqueue Apple jobs unless demand passes minimum popularity and confidence thresholds.
- Do not enqueue jobs for suspicious demand.
- Do not enqueue jobs when provider circuit is open.
- Do not enqueue jobs when provider budget is exhausted.

# Provider Budget Enforcement

Budgets must be checked in two places.

## Scheduler

The scheduler should avoid enqueueing jobs that are not currently budget-eligible.

## Worker

The worker must re-check provider budget immediately before making a third-party call.

This prevents races when multiple workers are running.

---

# Circuit Breaker

Each third-party provider must have a circuit breaker.

## Circuit States

```text
Closed
Open
HalfOpen
```

## Behaviour

```text
Closed:
  provider calls allowed

Open:
  provider calls blocked

HalfOpen:
  allow one test call
```

Open circuit when:

- repeated 429 responses
- repeated 5xx responses
- authentication failures
- malformed provider responses above threshold

Do not open circuit for normal `not found` responses.

---

Each provider/stage must declare its cost.

# Apple Enrichment Rules

Apple enrichment is allowed, but must be tightly controlled.

## AppleMusic Stage

Use when:

```text
provider AppleMusic is enabled
known ISRC exists
priority score >= AppleMusic minimum
provider budget is available
provider circuit is closed
```

Behaviour:

```text
ISRC
  ↓
Apple Music lookup by ISRC
  ↓
candidate Apple song IDs
  ↓
validate metadata where available
  ↓
store mapping
```

## ITunesSearch Stage

Use only when:

```text
AppleMusic failed or is unavailable
no Apple ID found from open metadata
artist/title confidence is high
demand is popular enough
priority score >= ITunesSearch minimum
provider budget is available
provider circuit is closed
```

Behaviour:

```text
artist + title
  ↓
iTunes Search API
  ↓
candidate results
  ↓
local validation by artist/title/duration
  ↓
store mapping only if confidence is high
```

## Apple Safety Rules

- Never call Apple from public API request path.
- Never call Apple for suspicious demand.
- Never call Apple for one-off low-demand queries unless manually overridden.
- Persist successful Apple mappings permanently.
- Persist negative results with retry-after.
- Keep Apple calls below configured budget.
- Treat 429 as a strong circuit-breaker signal.

# Queue Design

Use Azure Service Bus.

## Queues

```text
enrich-local
enrich-musicbrainz-api
enrich-apple-music
enrich-itunes-search
```

Each queue has:

- dead-letter queue
- max delivery count
- lock duration appropriate to provider
- duplicate detection where practical

Azure Service Bus does not provide native arbitrary priority ordering in a single queue. Use separate queues by stage/provider and have the scheduler enqueue only the highest priority eligible work.

Workers should process cheaper queues with higher concurrency and expensive queues with lower concurrency.

# Worker Processing Flow

```text
Receive job
  ↓
Load demand
  ↓
Check job idempotency
  ↓
Check demand still unresolved
  ↓
Check provider enabled
  ↓
Check circuit state
  ↓
Check provider budget
  ↓
Run enrichment stage
  ↓
Store attempt
  ↓
If resolved:
    store mapping
    update query cache
    update search index
    mark demand resolved
  ↓
If partial:
    update demand with new identifiers
    schedule next stage
  ↓
If not found:
    set next eligible time
  ↓
If retryable:
    requeue with delay
```

---

# Idempotency

Workers must be safe to retry.

## Rules

- Same job can be processed more than once.
- Mapping writes must be upserts.
- Attempt records must be append-only.
- Demand status updates must be conditional where possible.
- Provider budget increment must happen before provider call and be recorded.
- Failed jobs must not corrupt mappings.

---

# Retry Policy

## Local Failures

```text
retry quickly
max 3 attempts
```

## MusicBrainz API

```text
retry after minutes/hours
respect Retry-After if provided
```

## AppleMusic

```text
retry cautiously
respect Retry-After if provided
back off on 429 or 5xx
```

## ITunesSearch

```text
retry cautiously
back off aggressively on 429
```

## Permanent Negative Cache

If a provider confidently returns no result:

```text
do not retry for 7 to 30 days
```

The duration should be configurable per provider.

# Application Test Cases

Required tests:

1. Scheduler prioritises higher demand over lower demand.
2. Scheduler prefers local/open-data stages before Apple.
3. Scheduler does not enqueue suspicious demand.
4. Scheduler does not enqueue provider job when provider disabled.
5. Scheduler does not enqueue provider job when budget exhausted.
6. Scheduler does not enqueue Apple job below minimum priority.
7. Scheduler enqueues AppleMusic job when ISRC exists and priority is high enough.
8. Scheduler enqueues ITunesSearch only after cheaper stages fail.
9. Worker re-checks provider budget before provider call.
10. Worker records attempt for every job.
11. Worker stores mapping when enrichment resolves.
12. Worker marks demand resolved when mapping is found.
13. Worker schedules next stage when partial data is found.
14. Worker backs off on provider 429.
15. Worker opens circuit after repeated provider failures.
16. Worker does not open circuit for normal not-found.
17. Worker is idempotent when the same job is processed twice.

# Contract-Tested Fakes

Use parameterised tests where one test method covers fake and real implementation.

Example:

```csharp
public sealed class ProviderBudgetStoreContractTests
{
    public static IEnumerable<object[]> Modes =>
    [
        [StorageMode.Fake],
        [StorageMode.AzureTable]
    ];

    [Theory]
    [MemberData(nameof(Modes))]
    public async Task Budget_Is_Incremented_And_Limit_Is_Respected(
        StorageMode mode)
    {
        var env = ProviderBudgetTestEnvironment.Create(mode);

        var provider = ProviderName.From("AppleMusic");

        var policy = ProviderRateLimitPolicy.Create(
            maxPerMinute: 1,
            maxPerHour: 10,
            maxPerDay: 100);

        var first = await env.Budgets.TryConsumeAsync(
            provider,
            policy,
            CancellationToken.None);

        var second = await env.Budgets.TryConsumeAsync(
            provider,
            policy,
            CancellationToken.None);

        first.Allowed.Should().BeTrue();
        second.Allowed.Should().BeFalse();
    }
}
```

# Acceptance Criteria

The implementation is complete when:

- a background enrichment worker project exists
- scheduler reads unresolved demand
- scheduler calculates priority
- scheduler prefers cheaper stages before Apple
- scheduler prioritises popular songs first
- MusicBrainz local dataset enrichment stage exists
- MusicBrainz API enrichment stage exists
- AppleMusic enrichment stage exists
- ITunesSearch enrichment stage exists
- Spotify is not implemented initially
- each third-party provider has configurable rate limits
- provider budgets are enforced by scheduler and worker
- provider circuit breakers exist
- Apple enrichment is never called from the public API path
- Apple enrichment only runs for eligible high-priority demand
- successful mappings are persisted
- query cache/search index are updated after enrichment
- enrichment attempts are recorded
- retry and negative-cache behaviour exists
- Application Insights metrics/logs are emitted
- tests use custom fakes
- fakes are contract-tested against real adapters
- no mocking framework is used by default
- solution builds and tests pass with:

```text
dotnet test
```
