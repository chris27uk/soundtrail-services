# Track Discovery Service Redesign

## Intent

Replace the current enrichment-centric design with a business-focused track discovery design that is simpler to reason about, better aligned to the product goals, and explicit about provider safety.

This document supersedes the current mental model of:

- `ResolutionDemand`
- `EnrichmentScheduler`
- `EnrichmentJob`
- `EnrichmentWorker`
- `EnrichmentStage`
- `ProviderBudget`
- `ConcurrencyLease`

with the business-first model of:

- `TrackRequest`
- `DiscoveryPlanner`
- `DiscoveryTask`
- `DiscoveryAgent`
- `DiscoveryStep`
- `DiscoveryAllowance`
- `DiscoveryPermit`

It preserves the current strengths:

- feature-folder organisation
- project parity with the API split
- asynchronous discovery
- Azure-native deployment

## Business Objectives

The discovery service exists to answer one question:

> Is it worthwhile to spend provider budget trying to discover this track now?

The service must be:

- simple
- observable
- idempotent
- local-first
- cost-aware
- provider-safe

## Core Principles

### DemandDrivenDiscovery

- Unresolved searches create `TrackRequest` records.
- Public requests must never directly trigger provider calls.
- Discovery occurs asynchronously.

### ProviderProtection

- Provider calls require a `DiscoveryPermit`.
- Provider calls require a `DiscoveryAllowance`.
- Provider calls require a closed circuit.

### PopularityFirst

- Popular tracks are discovered before unpopular tracks.
- High trust requests are prioritised.
- Suspicious requests are deprioritised or ignored.

## Service Split

The design keeps the current split between request-path concerns and background processing concerns.

```text
src/
  Soundtrail.Services.Api/
    Features/
      Search/
      Health/
    Infrastructure/
      TableStorage/
      Search/

  Soundtrail.Services.Discovery/
    Features/
      TrackRequests/
      DiscoveryPlanning/
      DiscoveryExecution/
      ProviderAccess/
      TrackCatalogue/
      SearchMemory/
      MusicBrainz/
      AppleMusic/

  Soundtrail.Services.Discovery.Worker/
    Infrastructure/
      Hosting/
      Messaging/
      TableStorage/
      BlobStorage/
      Http/
```

`Soundtrail.Services.Api` is responsible for recording demand and serving local results.

`Soundtrail.Services.Discovery` owns the domain model and use cases for planning and executing discovery.

`Soundtrail.Services.Discovery.Worker` hosts planner and agent processes plus Azure-backed adapters.

## Responsibility Boundaries

### DiscoveryPlanner

Primary responsibility:

- decide whether discovery work is worthwhile

Allowed responsibilities:

- priority calculation
- popularity scoring
- risk filtering
- discovery scheduling

Forbidden responsibilities:

- Apple lookup logic
- MusicBrainz lookup logic
- provider-specific heuristics

The planner does not resolve tracks. It only decides if a request deserves a discovery attempt.

### DiscoveryAgent

Primary responsibility:

- execute discovery work

Responsibilities:

- load `TrackRequest`
- execute `DiscoveryChain`
- persist results
- update `TrackCatalogue`
- update `SearchMemory`

The agent is the executor and final authority for whether a provider call is actually allowed at runtime.

## Domain Model

### TrackRequest

Represents accumulated evidence that users want a track we cannot currently satisfy from local data.

Suggested shape:

```csharp
public sealed record TrackRequest(
    QueryId TrackRequestId,
    NormalizedSearchQuery Query,
    int RequestCount,
    int DistinctInstallCount,
    int DistinctIpHashCount,
    int HighestTrustLevelSeen,
    int RiskScore,
    TrackTitle? BestKnownTitle,
    ArtistName? BestKnownArtist,
    Isrc? BestKnownIsrc,
    Mbid? BestKnownMbid,
    TrackRequestStatus Status,
    DateTimeOffset FirstSeenAt,
    DateTimeOffset LastSeenAt,
    DateTimeOffset? NextEligibleAt,
    IReadOnlyCollection<DiscoveryStepName> AttemptedSteps,
    int FailureCount);
```

Key rules:

- multiple unresolved searches for the same normalized query update one `TrackRequest`
- repeated requests increase priority evidence
- suspicious requests may be ignored entirely
- the request is the unit of planning

### TrackCatalogue

Represents locally trusted track identities and mappings.

Responsibilities:

- store discovered identifiers
- answer local search and lookup flows
- support idempotent upserts from discovery results

### SearchMemory

Represents query-oriented local memory.

Responsibilities:

- remember successful query-to-result associations
- reduce repeat public misses
- refresh when `TrackCatalogue` improves

### DiscoveryStepName

`DiscoveryStepName` still exists, but only as metadata.

It is useful for:

- attempt history
- telemetry
- diagnostics
- test assertions

It is not the orchestration mechanism.

## Planning Model

The planner should work from `TrackRequest` records rather than from named stages.

### Planner Input

- unresolved or partially satisfied `TrackRequest` records
- current time
- provider policy snapshots
- circuit state

### Planner Output

- `TrackDiscoveryCommand`

Suggested shape:

```csharp
public sealed record TrackDiscoveryCommand(
    string CommandId,
    QueryId TrackRequestId,
    NormalizedSearchQuery Query,
    int PriorityScore,
    DateTimeOffset NotBefore,
    DateTimeOffset CreatedAt,
    string CorrelationId);
```

### Planner Rules

- only consider requests whose `NextEligibleAt` has passed
- ignore requests with risk above the reject threshold
- deprioritise requests with medium risk
- score popularity before provider cost
- score trust before provider cost
- do not embed provider-specific lookup logic
- do not assume concurrency is available
- only use provider availability checks as scheduling hints
- do not choose a named discovery step

### Priority Calculation

The priority model should remain explicit and inspectable.

Suggested score inputs:

- total request count
- distinct installs
- distinct IP hashes
- highest trust level
- known identifiers already present
- prior failed attempts
- risk score
- provider cost penalty

The score should be deterministic so that a given request is explainable in logs and tests.

## Discovery Execution Model

Discovery is implemented as a chain of responsibility.

The planner schedules "attempt discovery for this `TrackRequest`".

The agent owns the ordered chain and runs it from the beginning each time.

```text
LocalCatalogueStep
  → MusicBrainzDatasetStep
  → MusicBrainzApiStep
  → AppleMusicByIsrcStep
  → ITunesSearchStep
```

Each step:

- decides whether it can run
- may improve the candidate
- may resolve the track
- may stop the chain
- must not contain rate-limiting logic

This means discovery flow is driven by current request state and candidate state, not by a preselected step name.

Cheap local steps are expected to be fast and idempotent, so restarting the chain is acceptable and keeps orchestration simple.

### Important Rule

Discovery steps must not call providers directly.

Provider access must occur through `ProviderCallGate`.

This is the main structural change from the current design.

Today the step implementation and provider client are effectively fused in the same abstraction. In the redesign:

- a `DiscoveryStep` decides intent
- `ProviderCallGate` decides whether a remote call is legal
- a provider client performs the actual HTTP call only after the gate succeeds

### DiscoveryChain Contract

Suggested shape:

```csharp
public interface IDiscoveryStep
{
    DiscoveryStepName Name { get; }

    bool CanRun(TrackRequest request, DiscoveryContext context);

    Task<DiscoveryStepResult> RunAsync(
        TrackRequest request,
        DiscoveryContext context,
        CancellationToken ct);
}
```

Suggested chain contract:

```csharp
public interface IDiscoveryChain
{
    Task<DiscoveryChainResult> RunAsync(
        TrackRequest request,
        CancellationToken ct);
}
```

`DiscoveryContext` should carry:

- currently known candidate state
- read-only local projections
- provider gate
- correlation and telemetry context

`DiscoveryStepResult` should allow:

- no change
- candidate improved
- resolved
- retry later
- step skipped

`DiscoveryChainResult` should allow:

- resolved
- exhausted
- retry later
- blocked

## ProviderCallGate

All provider access must flow through a single business abstraction.

```csharp
public interface IProviderCallGate
{
    Task<ProviderCallDecision<T>> RunAsync<T>(
        ProviderName provider,
        Func<CancellationToken, Task<T>> action,
        CancellationToken ct);
}
```

Before executing `action`, the gate must require:

- `DiscoveryPermit`
- `DiscoveryAllowance`
- `CircuitClosed`

If any condition fails:

- the call does not happen
- the result is returned as a business outcome
- the step decides whether to skip or retry later

### DiscoveryPermit

Purpose:

- global distributed concurrency control

Rules:

- concurrency limits are global
- local semaphores are insufficient
- use distributed permits
- store permits in Azure Table Storage

Example provider limits:

- `MusicBrainz` max concurrent: `3`
- `AppleMusic` max concurrent: `1`
- `ITunesSearch` max concurrent: `1`

### DiscoveryAllowance

Purpose:

- control provider consumption separately from concurrency

Dimensions:

- per minute
- per hour
- per day

The allowance model must answer:

- can we afford this call now?
- if not, when should this request become eligible again?

### Circuit State

States:

- `Closed`
- `Open`
- `HalfOpen`

Open conditions:

- repeated `429` responses
- repeated `5xx` responses
- authentication failures

Ignored conditions:

- `404` / not found

The circuit is part of provider protection, not part of step logic.

## Recommended Execution Flow

```text
DiscoveryAgent receives TrackDiscoveryCommand
  ↓
Load TrackRequest
  ↓
Build DiscoveryContext
  ↓
Run DiscoveryChain from the first step
  ↓
For remote steps, call ProviderCallGate
  ↓
Persist attempt + result
  ↓
Update TrackCatalogue
  ↓
Update SearchMemory
  ↓
Update TrackRequest state
```

## Observability

The redesign should make decisions explainable.

Every planner and agent run should emit structured logs containing:

- `TrackRequestId`
- query
- priority score
- skip reason
- step name
- provider name
- permit result
- allowance result
- circuit state
- discovery outcome
- next eligible time

Recommended metrics:

- track requests created
- track requests updated
- planner candidates considered
- planner commands emitted
- discovery steps attempted
- discovery steps resolved
- provider calls allowed
- provider calls blocked by permit
- provider calls blocked by allowance
- provider calls blocked by circuit
- suspicious requests ignored

## Testing Strategy

The service should continue to follow the current testing style:

- outside-in TDD
- custom fakes
- contract-tested fakes
- scenario test environments

Rules:

- no mocking framework by default
- verify behaviour rather than interactions
- fakes must be contract-tested against real implementations

Suggested test layers:

### API scenario tests

- unresolved search creates `TrackRequest`
- unresolved search does not call providers
- repeated unresolved searches increase request evidence

### Planner scenario tests

- popular requests produce commands before unpopular requests
- high-trust requests outrank anonymous requests
- suspicious requests are skipped or heavily penalised
- open circuits suppress provider-backed planning
- planner does not depend on stage progression

### Agent scenario tests

- chain restarts from the first step safely
- local steps execute without provider gate
- remote steps require permit, allowance, and closed circuit
- blocked provider access returns retry-later without calling the client
- successful discovery updates both `TrackCatalogue` and `SearchMemory`
- repeated `429` or `5xx` responses open the circuit

### Contract tests

- Azure Table permit store
- Azure Table allowance store
- Azure Table circuit store
- Service Bus command queue
- provider client fakes vs real HTTP adapters

## Migration Strategy

This redesign can be introduced incrementally.

### Phase 1: Rename the model

- introduce `TrackRequest` alongside `ResolutionDemand`
- introduce `DiscoveryTask` / `TrackDiscoveryCommand` alongside `EnrichmentJob`
- introduce `DiscoveryStepName` alongside `EnrichmentStage`
- update logs, options, and telemetry to use business language first

### Phase 2: Split planning from execution more clearly

- keep current queue and worker process
- replace `EnrichmentScheduler` with `DiscoveryPlanner`
- make planner emit business commands rather than provider jobs
- remove step selection from planner output
- move step ordering into `DiscoveryChain`

### Phase 3: Introduce `ProviderCallGate`

- remove direct provider access from step classes
- move permit, allowance, and circuit checks into one gate
- convert provider steps to use the gate

### Phase 4: Replace enrichment vocabulary completely

- remove old names
- rename projects from `Enrichment` to `Discovery`
- keep feature-folder parity with the API

## Why This Design Is Simpler

This design simplifies the service in four ways:

1. It uses business language at the top level.
2. It gives the planner only one job: decide whether work is worth doing.
3. It gives the agent only one job: execute a discovery chain safely.
4. It centralises provider protection into a single gate instead of scattering it across scheduler, worker, and step abstractions.

It also removes the need for a shared stage vocabulary across planner, queue, worker, and steps, which is where much of the current orchestration complexity comes from.

## Recommended Next Implementation Slice

The lowest-risk next step is:

1. add the new domain names in code without deleting the old ones
2. introduce `DiscoveryChain` as the primary execution abstraction
3. introduce `ProviderCallGate`
4. refactor one remote step, preferably `MusicBrainzApi`, to use the gate

That sequence keeps behaviour stable while moving the architecture toward the intended model.
