# Music Resolver API Specification

> Historical note: [1.0.SoundtrailServicesCanonicalSpecification.md](/Users/chris/Developer/Soundtrail.Services/specs/1.0.SoundtrailServicesCanonicalSpecification.md) is the current authoritative spec. Use this document as supporting context only where it does not conflict with the canonical specification.

## Goal

Build a public C# API that resolves music search queries to known identifiers, especially Apple/iTunes IDs, without acting as a live proxy to Apple, Spotify, or other third-party services.

The API must be:

- cache-first
- abuse-resistant
- responsive for end users
- simple to reason about
- structured using a lightweight hexagonal architecture
- tested outside-in TDD
- based on custom fakes instead of mocking frameworks

---

# Core Principles

## Public requests must never directly call third-party APIs

Public traffic must only:

- query local indexes
- query local caches
- create demand signals

Third-party enrichment must happen asynchronously and under controlled budgets.

---

## Responsive UI behaviour

The API should return immediately even when results are unknown.

Example flow:

```text
User searches
  ↓
API returns local results immediately
  ↓
If unresolved:
  - create deduplicated demand signal
  - return pending state
  ↓
Background workers enrich later
```

---

## Build a local resolver graph over time

The system should converge toward:

```text
ISRC ↔ MBID ↔ Apple ID ↔ Spotify ID
```

stored locally.

The long-term goal is minimizing third-party traffic.

---

- no mocking frameworks by default

Checks:

- music catalog lookup store
- search index

Returns:

```json

  "status": "ready"
}
```

---

## GET /search

### Query Parameters

```text
q       required string
limit   optional int default 10 max 25
```

### Behaviour

1. Normalize query.
2. Check query cache.
3. Search local search index.
4. Return best known local results.
5. If weak/no result:
   - record demand signal
   - return pending/partial response
6. Never call third-party APIs directly.

---

# Test Style

Use the TestEnvironment style described in Chris Green’s article.

Avoid builder-heavy tests.

Prefer:

```csharp
var env = SearchMusicTestEnvironment.WithKnownTrack();

var sut = env.CreateHandler();

var result = await sut.Handle(
    env.SearchForKnownTrack());

result.Status.Should().Be(
    ResolutionStatus.Resolved);
```

Avoid:

```csharp
new SearchMusicHandlerBuilder()
    .WithCache(...)
    .WithSearch(...)
    .Build();
```

# Contract-Tested Fakes

Every fake used in application tests must be contract-tested against the real implementation.

Use one parameterised test method that runs against:

- the fake implementation
- the real implementation

---

## Example Contract Test

```csharp
public sealed class QueryCacheContractTests
{
    public static IEnumerable<object[]> Modes =>
    [
        [StorageMode.Fake],
        [StorageMode.AzureTable]
    ];

    [Theory]
    [MemberData(nameof(Modes))]
    public async Task Stored_Response_Can_Be_Read_Back(
        StorageMode mode)
    {
        var env = QueryCacheTestEnvironment.Create(mode);

        var query = SearchQuery.From("mr brightside");

        var response =
            SearchMusicResponse.Resolved(
                query,
                [KnownTracks.MrBrightside()]);

        await env.Cache.StoreAsync(
            query,
            response,
            TimeSpan.FromHours(1),
            CancellationToken.None);

        var actual =
            await env.Cache.GetAsync(
                query,
                CancellationToken.None);

        actual.Should().BeEquivalentTo(response);
    }
}
```

# Suggested Unit Coverage

The current unit layer should prove the business decisions in the API application layer, not just the shape of the response payload.

Suggested API unit scenarios:

- known local result returns `Resolved`
- known local result does not enqueue lookup work
- unknown query returns `Pending`
- unknown query enqueues one lookup request
- lookup request uses the normalized query value
- lookup request gets a correlation id
- lookup request records an occurrence timestamp
- minimum confidence keeps qualifying results
- minimum confidence removes weaker results and falls back to pending
- limit trims local results to the requested count
- query normalization handles case, punctuation, and repeated whitespace
- default limit is used when omitted
- invalid limits are rejected at the request model boundary
- minimum confidence boundary values are accepted and rejected correctly

Suggested policy/value-object unit scenarios:

- `NormalizedSearchQuery` lowercases input
- `NormalizedSearchQuery` removes punctuation without losing token boundaries
- `NormalizedSearchQuery` collapses repeated whitespace
- `Limit` defaults to `10`
- `Limit` accepts `1` and `25`
- `Limit` rejects `0` and `26`
- `ConfidenceScore` accepts `0` and `1`
- `ConfidenceScore` rejects values below `0` or above `1`

Notes on emphasis:

- prefer one business scenario per test over repeating all response fields across many tests
- keep contract tests for fakes, but use unit tests to lock down normalization and threshold rules
- prioritize tests that would fail if business rules changed, not just if sample data changed

The implementation is complete when:

- API starts successfully.
- Health endpoints exist.
- Search endpoint returns local/cached data only.
- Unknown queries create demand signals.
- Public endpoints never directly call third-party services.
- Application layer has no infrastructure references.
- Tests use custom fakes rather than mocks.
- Fakes are contract-tested against real adapters.
- Tests use the TestEnvironment pattern.
