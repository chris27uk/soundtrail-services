# Enrichment Orchestrator Plan

> Proposal note: [1.0.SoundtrailServicesCanonicalSpecification.md](/Users/chris/Developer/Soundtrail.Services/specs/1.0.SoundtrailServicesCanonicalSpecification.md) remains the current authoritative spec. This document is a proposed next-step plan for orchestration, event flow, and projection ownership.

## Purpose

This document defines the proposed direction for evolving the current enrichment scheduler into an orchestration model that:

- keeps provider execution out of the public API path
- keeps workers operationally simple
- makes orchestration decisions explicit
- makes state transitions explicit through domain events
- projects usable track state into the API-facing Raven store

It is intentionally a plan, not yet the canonical contract.

## Why Change

The current solution has grown beyond pure scheduling.

It now has to coordinate:

- unresolved search demand
- provider-specific execution commands
- worker responses
- application of enrichment results
- follow-up provider verification
- readiness of API-facing playable track data

`Scheduler` no longer fully describes that responsibility. The planned destination is an `Orchestrator`, but the rename should follow the model change rather than lead it.

## Business Outcome

The product does not gain value from metadata alone.

A track is only meaningfully useful when:

1. canonical identity and metadata are trusted
2. at least one playback-capable provider reference has been verified

`MusicBrainz` is the canonical metadata authority.

`Apple` and `YouTube Music` are playback-capable provider authorities for their own references.

`MusicBrainz` may discover cross-provider references, but those references are not trusted for playback until verified by the owning provider.

## Core Principles

### Canonical First

- `MusicBrainz` establishes canonical metadata and identity.
- downstream provider verification should not run until canonical identity is strong enough

### Explicit Intent

The orchestrator should issue commands that describe the enrichment intent, not just the target provider.

Bad:

- `ExecuteAppleLookup`

Better:

- `VerifyApplePlaybackReference`

### Verified Playback

- a `MusicBrainz`-discovered Apple or YouTube Music reference may be stored as discovered
- it does not make a track playable
- a track becomes playable only when a playback-capable provider verifies its own reference

### Projection Separation

- orchestration facts are the source of workflow truth
- the API Raven store is a projection for query and serving
- those two concerns may share physical infrastructure, but should remain conceptually separate

### Workers Stay Dumb

Workers should:

- receive explicit commands
- execute one provider-specific intent
- emit explicit results

Workers should not:

- choose providers
- decide completion policy
- write API-facing projection state directly

## Aggregate Boundary

The proposed aggregate boundary is one stream per `MusicCatalogId`.

This aggregate represents the orchestration state of one track identity as it moves from:

- unresolved
- canonically resolved
- playback-reference discovered
- playback-reference verified
- playable

The aggregate should not model general API search concerns or HTTP request concerns.

It should model only the orchestration lifecycle for enrichment of one track.

## Proposed Aggregate Responsibilities

The aggregate should decide:

- whether canonical metadata has been established
- which provider references are only discovered
- which provider references are verified
- whether the track is now playable
- whether additional provider verification work is still needed
- whether the orchestration is complete enough for current business purposes

It should not decide:

- HTTP response formatting
- provider client implementation details
- transport wiring
- Raven document shapes for serving the public API

## Lifecycle Milestones

### `Unresolved`

- no trusted canonical identity
- no verified playback references

### `CanonicalResolved`

- `MusicBrainz` has established canonical metadata
- cross-provider references may or may not have been discovered
- the track is not yet playable unless a playback-capable provider reference is later verified

### `PlayableResolved`

- canonical metadata is established
- at least one playback-capable provider reference is verified

This is the minimum meaningful completion milestone for current business intent.

## Proposed Commands

The orchestrator should emit explicit commands with intent in their names.

### Canonical Commands

- `ResolveCanonicalMetadata`

Purpose:

- ask `MusicBrainz` to establish canonical identity and metadata

### Verification Commands

- `VerifyApplePlaybackReference`
- `VerifyYouTubeMusicPlaybackReference`

Purpose:

- verify a playback-capable reference with the owning provider

### Future Optional Commands

- `DiscoverApplePlaybackReference`
- `DiscoverYouTubeMusicPlaybackReference`

These should only exist if we later decide there is value in direct provider discovery when `MusicBrainz` has not yielded a usable cross-provider reference.

For the current plan, verification-oriented commands are preferred.

## Proposed Events

The orchestrator should persist explicit facts.

### Intake And Planning

- `TrackLookupRequested`
- `CanonicalResolutionRequested`
- `ApplePlaybackVerificationRequested`
- `YouTubeMusicPlaybackVerificationRequested`

### Canonical Results

- `CanonicalMetadataResolved`
- `CanonicalMetadataResolutionFailed`

### Reference Discovery

- `ProviderReferenceDiscovered`

This event means:

- a reference was discovered
- it is not yet verified unless the source provider owns it

### Reference Verification

- `ProviderReferenceVerified`
- `ProviderReferenceVerificationFailed`

### Milestone Events

- `TrackBecamePlayable`
- `EnrichmentCompleted`

`EnrichmentCompleted` should mean:

- canonical metadata exists
- at least one playback-capable provider reference is verified

It should not mean:

- every provider has been tried
- every provider has succeeded

## Trust Rules

### `MusicBrainz`

Trusted for:

- canonical metadata
- canonical identity
- cross-provider reference discovery

Not trusted for:

- final verification of Apple playback references
- final verification of YouTube Music playback references

### `Apple`

Trusted for:

- verification of Apple playback references

Not trusted for:

- replacing canonical metadata by default

### `YouTube Music`

Trusted for:

- verification of YouTube Music playback references

Not trusted for:

- replacing canonical metadata by default

## Completion Rule

The orchestrator should not stop at `MusicBrainz` metadata alone.

Completion for the current business intent requires:

1. canonical metadata has been resolved
2. at least one playback-capable provider reference has been verified

If `MusicBrainz` returns Apple or YouTube Music references, the orchestrator should:

- store them as discovered
- issue verification work where appropriate
- only promote the track to playable after verification succeeds

## Projection Rule

The API Raven store should be treated as a projection of orchestration facts.

The projection should become queryable once canonical metadata exists, but it should distinguish between:

- canonical-only track state
- discovered provider references
- verified provider references
- playable status

This lets the product:

- show trusted canonical information early if desired
- avoid presenting unverified playback links as playable

## Projection Ownership

Ownership should be split conceptually as follows:

- the orchestrator owns workflow facts and enrichment-state decisions
- the API owns the query-serving shape of the track projection

This means:

- the orchestrator should emit and persist events
- projection logic should translate those facts into API-facing Raven documents
- the API Raven store is not the aggregate itself

Whether the projection updater lives physically in the orchestrator host or a dedicated projection component is an implementation choice.

For the first cut, it is acceptable for the orchestrator host to update the projection, as long as the conceptual split remains clear.

## Event Sourcing Scope

The proposed direction is event-oriented first, not full event sourcing everywhere immediately.

Recommended first step:

- define explicit commands
- define explicit events
- append orchestration facts
- update the API projection from those facts

Only after the event boundaries prove stable should the team decide whether:

- the orchestration aggregate should be fully reconstructed from historical events at load time
- all current mutable orchestration state should be replaced by a pure event-sourced model

This keeps the transition incremental and lowers the risk of over-architecting too early.

## Proposed Rename Path

Destination naming:

- `Soundtrail.Services.Enrichment.Orchestrator`

Rename only when these are true:

1. the host owns both work issuance and response application
2. the command model is explicit and intent-based
3. event flow is the dominant mental model

Until then:

- `Scheduler` may remain the physical project name
- `orchestration` language can still be used inside the proposal and future code

## Implementation Phases

### Phase 1

- define command intents
- define orchestration events
- keep current mutable state where necessary
- project canonical and provider-reference state into the API Raven store

### Phase 2

- add event log persistence
- move apply logic from mutable ad hoc transitions to event-driven transitions
- introduce aggregate state object over orchestration lifecycle

### Phase 3

- evaluate whether full event-sourced aggregate reconstruction is paying its way
- rename `Scheduler` to `Orchestrator` if the model has genuinely shifted

## Open Questions

- Should canonical-only tracks be visible to end users before a playable reference is verified, or only internally queryable?
- Does a failed Apple verification automatically trigger YouTube Music verification, or is that a separate orchestration policy?
- Should there be one orchestration stream per `MusicCatalogId`, or separate sub-streams per provider intent under a shared root?
- Where should projection updating live operationally once the system is scaled further?

## Proposed Next Step

Before renaming projects or introducing a full event-sourced repository pattern, the next design step should be:

1. agree the command list
2. agree the event list
3. agree the completion rule
4. agree the projection shape for canonical, discovered, verified, and playable states

Only after that agreement should implementation proceed.
