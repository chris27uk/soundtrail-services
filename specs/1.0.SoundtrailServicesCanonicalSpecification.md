# Soundtrail Services Canonical Specification

## Status

This document is the canonical specification for the repository.

It supersedes the older intent documents as the primary source of truth for new work in this codebase.

Older specification files remain useful as historical context and for future ideas, but they are not authoritative when they conflict with this document.

## Reconciliation Rule

When an older spec conflicts with the current project:

- prefer the current project structure, naming, and behavior
- unless the current implementation clearly violates the business intent described here

The business intent in this document takes precedence over historical implementation sketches, folder examples, and obsolete terminology.

---

## Product Aim

Soundtrail Services exists to resolve music search queries to trusted local track identities without making the public request path depend on live third-party providers.

The system should:

- return local results quickly
- accumulate evidence for unresolved searches
- schedule discovery work asynchronously
- spend provider budget carefully
- improve local knowledge over time

The long-term objective is a stronger local music catalogue with richer identifier mappings, especially Apple/iTunes identifiers, while keeping public traffic provider-safe.

For playback-oriented enrichment, the current business intent is specifically to attach proprietary provider ids that allow downstream playback on Sonos.

---

## Business Objectives

The service must be:

- local-first
- responsive
- idempotent
- cost-aware
- provider-safe
- simple to reason about
- observable through explicit state and deterministic rules
- testable with outside-in TDD and custom fakes

## Core Business Rules

### Public Request Path Safety

Public API requests must never directly call Apple, Spotify, MusicBrainz, or any other third-party provider.

Public traffic may only:

- search local indexed data
- use local cached knowledge where implemented
- emit asynchronous discovery demand

### Immediate User Response

Search requests should return immediately.

If the system has good local results, it returns them.

If the system does not have acceptable local results, it returns a pending response and records enough information for background discovery planning.

### Discovery Is Evidence-Driven

Unresolved searches are signals, not direct provider work orders.

Repeated requests, trust level, and risk level influence whether discovery is worth attempting and how urgently it should be scheduled.

### Local Knowledge Should Improve Over Time

The system should converge toward richer local mappings between:

- normalized user queries
- local track representations
- music catalogue ids
- external ids such as ISRC, MBID, Apple ID, and optionally Spotify ID when locally known

Spotify may exist as a stored identifier on local track records, but Spotify provider integration is not part of the current scheduled discovery implementation.

### Provider Hierarchy

The providers do not play the same business role.

- `MusicBrainz` is the canonical metadata discovery provider
- `Apple` is a proprietary-id enrichment provider
- `YouTube Music` is a proprietary-id enrichment provider

`MusicBrainz` should be treated as the source of truth for most metadata, matching confidence, and shared identity establishment.

`Apple` and `YouTube Music` should primarily add their own proprietary ids to already-known tracks so that downstream playback systems such as Sonos can use them.

`Apple` and `YouTube Music` are not canonical metadata authorities in the current intended design.

## Current Implemented Scope

The codebase currently implements:

- a public search API
- local Raven-backed search
- asynchronous enqueueing of unresolved search requests
- enrichment planning and scheduling logic
- just-in-time lookup scheduling for new requests
- backlog scheduling for accumulated candidates
- Raven-backed scheduler persistence and idempotency stores
- Azure Service Bus integration for request intake and scheduled command publication
- a first worker skeleton for provider-specific execution lanes

The codebase does not currently implement the full provider-execution pipeline described in older worker specs.

Not currently implemented end-to-end:

- provider budget enforcement
- provider circuit breakers
- MusicBrainz live enrichment execution
- Apple enrichment execution
- YouTube Music enrichment execution
- retry/dead-letter discovery execution stages
- a full search-memory or query-cache flow in the API request path
- the authentication and abuse-protection spec referenced by older docs

`Soundtrail.Services`

- core search use-case behavior
- track and identifier value types
- shared request/response models
- shared abstractions such as handlers and message contracts

- minimal HTTP endpoints
- Raven-backed local search adapter
- HTTP request validation and translation
- enqueueing unresolved lookups
- API host composition

`Soundtrail.Services.Enrichment`

- discovery planning rules
- lookup resolution policy
- priority policy
- ranked candidate model
- active-work and candidate-store abstractions

- scheduler host
- Wolverine listeners

- provider selection for downstream worker commands

- worker host
- provider-specific queue listeners
- execution idempotency
- provider execution wiring
- dumb execution flow driven by scheduler-issued commands

## Canonical API Behavior

### Health

`GET /health/live`

- returns `{ "status": "alive" }`

`GET /health/ready`

- currently checks readiness through `ITrackSearchPort`
- returns `200` with `{ "status": "ready" }` when the local search dependency is ready
- returns `503` when it is not ready

This is the current behavior even though older specs describe broader readiness checks.

### Search

`GET /search`

Supported query parameters:

- `q` required
- `limit` optional, default `10`, max `25`
- `minConfidence` optional, `0` to `1`

Current request-path flow:

1. validate request parameters
2. normalize the search query
3. search the local track index
4. optionally filter results by minimum confidence
5. if any acceptable local results remain, return `resolved`
6. otherwise enqueue a `LookupMusicRequest` and return `pending`

Current search behavior does not separately read a query cache before local search, even though earlier specs anticipated that step.

### Resolved Response

Current resolved responses include:

- `status`
- `source`
- `query`
- `results`

Each result may include:

- `title`
- `artist`
- `isrc`
- `mbid`
- `appleId`
- `spotifyId`
- `confidence`

### Pending Response

Current pending responses include:

- `status`
- `source`
- `query`
- `retryAfterSeconds`
- `results`

Older specs referenced `queryId` in pending responses. That is not current behavior and is not canonical unless deliberately introduced.

## Canonical Discovery And Scheduling Behavior

### Intake

When the API cannot satisfy a search request with acceptable local results, it emits a `LookupMusicRequest` containing:

- normalized query
- trust level
- risk score
- occurrence time
- correlation id

Current API behavior initializes:

- `TrustLevel = 0`
- `RiskScore = 0`

This is a valid current implementation, even though older documents assumed richer trust and abuse inputs.

### Just-In-Time Scheduling

`LookupMusicRequestHandler` is the current use case for handling unresolved lookup requests.

Its current business responsibilities are:

1. search the local music-catalog candidate source
2. resolve the best candidate using score and ambiguity rules
3. create or update a `RankedMusicCandidate`
4. determine whether the candidate deserves scheduling now
5. acquire active-work protection to avoid duplicate scheduling
6. return a high- or low-priority `LookupMusicCommand` when appropriate

If the local candidate search cannot produce an acceptable unambiguous `MusicCatalogId`, scheduling fails rather than falling through to provider execution.

This is intentional for the current scope.

### Backlog Scheduling

`DiscoveryBacklogScheduler` is the current batch planner.

Its responsibilities are:

- pull planning candidates from the ranked candidate store
- apply the priority policy
- skip ignored, resolved, suspicious, or not-yet-eligible candidates
- acquire active-work protection
- emit commands for schedulable candidates

### Provider Routing

The scheduler is responsible for deciding whether work should be sent to a provider and which provider should receive it.

That means:

- provider selection belongs to scheduling, not worker execution
- provider-specific command identities should be generated by the scheduler
- the worker should not infer which provider should run from ambiguous generic work items

This is especially important because `MusicBrainz` establishes canonical metadata, while `Apple` and `YouTube Music` are downstream enrichment providers for proprietary playback ids.

In practical terms, the intended discovery order is:

1. establish strong canonical identity through `MusicBrainz`
2. enqueue `Apple` enrichment when that identity is strong enough and worthwhile
3. enqueue `YouTube Music` enrichment when that identity is strong enough and worthwhile

### Priority Policy

The current priority rules are explicit and deterministic:

- non-pending candidates are ignored
- not-yet-eligible candidates are deferred
- suspicious candidates are ignored
- medium-risk candidates may still be scheduled, but only at low priority
- candidates with high trust or repeated demand are promoted to high priority
- otherwise candidates schedule at low priority

### Resolution Policy

The current candidate-resolution rules are also explicit:

- no matches means not found
- a top match below the minimum accepted score is rejected
- close competing strong matches are ambiguous
- a clear winner resolves to a `MusicCatalogId`

This policy is part of the current planning scope and is not yet a provider-execution policy.

### Idempotency

The scheduler must avoid duplicate concurrent work for the same music catalogue id.

Current code does this through `IActiveLookupWorkStore` and a time-bounded active reservation.

For worker execution idempotency, the command id is the execution identity.

That means:

- the scheduler is responsible for generating the correct provider-specific `CommandId`
- the worker idempotency store should only key on `CommandId`
- the worker should not need provider, correlation id, or timestamp to determine whether a command has already been processed

## Persistence And Infrastructure

### Current Canonical Infrastructure Choices

The current implementation uses:

- RavenDB for local search and scheduler persistence
- Azure Service Bus for asynchronous message flow
- Wolverine for listeners and message publication

This overrides earlier spec examples that assumed Azure Table Storage or SQLite FTS as the primary current implementation.

Those technologies may still be valid future alternatives, but they are not the canonical current design.

### Raven Usage Today

Raven currently stores:

- local track documents for search
- ranked music candidates
- active lookup work reservations
- worker execution receipts keyed by command id

Older specs mention query cache, demand records, and provider budgets as first-class persisted concepts. Those are not all currently implemented and should not be described as present behavior.

## Explicit Non-Goals For Current Scope

The following are not part of the current canonical implementation scope:

- live provider execution in the public API path
- Spotify provider integration
- speculative redesign terminology replacing current model names
- a full provider-budget and distributed-concurrency system
- claiming that Apple or MusicBrainz enrichment is already implemented end-to-end
- allowing worker handlers to perform provider-selection logic that belongs in scheduling

## Testing Canon

The codebase follows these testing expectations:

- outside-in TDD
- custom fakes over mocking frameworks
- sociable unit tests by default
- contract-style integration tests for fake-versus-real adapter parity
- scenario-led test naming

Unit tests should usually prove business rules through handlers, schedulers, and meaningful in-memory collaborators rather than through isolated policy-only tests, unless a standalone value type or pure rule genuinely benefits from solitary testing.

## Acceptance Criteria For The Current Project

The project is aligned with this canonical spec when:

- the public API serves local results without direct third-party calls
- unresolved searches emit asynchronous lookup requests
- scheduler logic prioritizes worthwhile discovery attempts
- duplicate scheduling is prevented through active-work protection
- local search and scheduling behavior remain deterministic and test-covered
- new work uses the current solution structure and canonical terminology
- future features do not claim capabilities that the codebase does not actually implement

---

## Future Evolution Rule

Future redesigns are allowed, including a move toward discovery-oriented naming or richer provider controls, but those changes should update this document first or as part of the same change.

Until then, this document defines the business intent and implementation boundaries for the repository.
